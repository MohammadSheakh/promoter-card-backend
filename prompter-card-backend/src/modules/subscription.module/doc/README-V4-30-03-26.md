## 🎯 Overview

**Purpose**: Manages subscription plans and user subscriptions with dual payment gateway support (Stripe for web business plans, RevenueCat for mobile individual plans).

**Business Value**: Enables monetization through flexible subscription tiers (Individual, Business Starter, Business Level 1, Business Level 2) with support for web and mobile platforms, free trials, auto-renewal, and admin management.

**Users**:
- **Individual Users** (Child/Patient Role) - Purchase and manage personal subscriptions via mobile apps (iOS/Android)
- **Business Users** (Business Role) - Purchase group/family subscriptions via web dashboard
- **Admins** - Manage subscription plans, view all subscriptions, create manual subscriptions

## 🏗️ Architecture Design

### Design Principles

1. **Hybrid Gateway Architecture** - Individual plans use RevenueCat (mobile IAP), Business plans use Stripe (web)
2. **Unified Schema** - Single `UserSubscription` collection stores subscriptions from both gateways
3. **Webhook-Driven Updates** - Real-time status synchronization via Stripe and RevenueCat webhooks
4. **Idempotency** - Webhook handlers prevent duplicate transactions with order ID checks

### Tech Stack
- **Database**: MongoDB (Mongoose) with pagination support
- **Cache**: Redis (cache-aside pattern for plan listings)
- **Payment Gateways**: Stripe (web), RevenueCat (mobile)
- **Validation**: Zod schemas
- **Auth**: JWT with role-based access control (TRole)
- **Scheduling**: Cron jobs for auto-renewal processing

## 🔒 Security Notes

> **⚠️ Compliance & Security Considerations**

### Payment Card Industry (PCI) Compliance
- ✅ **No Card Data Storage**: Card details never stored in database (handled by Stripe/RevenueCat)
- ✅ **Secure Webhooks**: HMAC-SHA256 signature verification for all webhook events
- ✅ **Tokenization**: Stripe customer IDs and RevenueCat user IDs used for references

### Data Protection
- ✅ **Soft Delete**: Subscriptions preserved with `isDeleted` flag (not permanently deleted)
- ✅ **Audit Trail**: PaymentTransaction records maintain complete payment history
- ✅ **Idempotency**: Webhook handlers prevent duplicate transactions via order ID checks

### Access Control
- ✅ **Role-Based Access**: 
  - `admin` - Full subscription plan management, manual subscription creation
  - `business` - Purchase business plans via Stripe
  - `child/individual` - Purchase individual plans via RevenueCat
  - `patient` - Start free trials, manage own subscriptions
- ✅ **User Isolation**: Users can only access their own subscriptions (auto-filtered by `userId`)

### Webhook Security
- ✅ **Signature Verification**: Both Stripe and RevenueCat webhooks verify HMAC signatures
- ✅ **Timing-Safe Comparison**: Prevents timing attacks on signature validation
- ✅ **Raw Body Middleware**: `express.raw()` positioned before `express.json()` for signature integrity

## 🗄️ Redis Caching

<details>
<summary>📦 <b>Click to view caching details</b></summary>

| Key Pattern | TTL | Use Case | Where to Invalidate |
|-------------|-----|----------|---------------------|
| `subscription:plans:active` | 10 min | Cache active subscription plans list | `createSubscriptionPlan()`, `updateById()` |
| `subscription:plans:all` | 10 min | Cache all subscription plans | `createSubscriptionPlan()`, `deleteById()` |
| `user:subscription:{userId}` | 10 min | Cache user's active subscription | `cancelSubscription()`, `createManualSubscription()` |
| `user:subscriptions:{userId}:list` | 10 min | Cache user's subscription history | `createManualSubscription()`, webhook handlers |

### Cache Strategy

**Pattern**: Cache-aside (lazy loading)

**On Read**:
1. Check Redis cache for key
2. If hit → return cached data
3. If miss → read from database
4. Cache result with TTL
5. Return data

**On Write**:
1. Perform database operation
2. Invalidate related cache keys
3. Return response

<details>

### BullMQ Queues

### Socket Events
**None** - Real-time updates via webhook-driven database changes

### Webhook Events (Async)

#### Stripe Webhooks (`/api/v1/stripe-webhook`)
- `checkout.session.completed` - Payment completed, create UserSubscription
- `invoice.payment_succeeded` - Recurring payment succeeded, update subscription
- `invoice.payment_failed` - Payment failed, set status to `past_due`
- `customer.subscription.created` - Subscription created in Stripe
- `customer.subscription.deleted` - Subscription cancelled in Stripe
- `customer.subscription.updated` - Subscription updated in Stripe

#### RevenueCat Webhooks (`/api/v1/revenuecat-webhook`)
- `INITIAL_PURCHASE` - User subscribes for first time → Create UserSubscription
- `RENEWAL` - Subscription renewed → Update dates, create PaymentTransaction
- `CANCELLATION` - User cancelled → Set `cancelledAtPeriodEnd`
- `EXPIRATION` - Subscription expired → Set status to `expired`
- `REFUND` - Refund processed → Set payment status, cancel subscription
- `BILLING_ISSUE` - Payment failed → Set status to `past_due`
- `SUBSCRIPTION` - General subscription update

### Cron Jobs (`userSubscription.cron.ts`)
- **Auto-Renewal Processing** - Daily check for subscriptions due for renewal
- **Expiration Processing** - Daily check for expired subscriptions
- **Trial Conversion** - Convert trial subscriptions to paid after trial period

## 🐛 Troubleshooting

<details>
<summary>🔧 <b>Click to view troubleshooting guides</b></summary>

### Issue: Webhook Signature Verification Failed

**Symptom**: Webhook returns 401 Unauthorized, logs show "Invalid webhook signature"

**Solution**: Verify webhook secret and middleware order

```bash
# Check webhook secret in .env
cat .env | grep REVENUECAT_WEBHOOK_SECRET
cat .env | grep STRIPE_WEBHOOK_SECRET

# Verify express.raw() is before express.json() in app.ts
grep -A 5 "revenuecat-webhook" src/app.ts
grep -A 5 "stripe-webhook" src/app.ts
```

**Fix**: Ensure middleware order:
```typescript
// ✅ CORRECT
app.post('/api/v1/revenuecat-webhook', express.raw({ type: 'application/json' }), handler);
app.use(express.json());

// ❌ WRONG
app.use(express.json());
app.post('/api/v1/revenuecat-webhook', handler);  // Will fail!
```

### Issue: User Already Used Free Trial

**Symptom**: User receives "User is not eligible for free trial" error

**Solution**: Check user.hasUsedFreeTrial flag

```bash
# Check user record
mongosh
db.User.findOne({ email: "user@example.com" }).projection({ hasUsedFreeTrial: 1 })

# If false and should be eligible (edge case), reset manually
db.User.updateOne(
  { email: "user@example.com" },
  { $set: { hasUsedFreeTrial: false } }
)
```

```bash
# Check backend logs
tail -f logs/app.log | grep "webhook"

# Check for duplicate order ID
mongosh
db.UserSubscription.find({ revenueCatOrderId: "order_123" })

db.UserSubscription.updateOne(
  { _id: "subscription_id" },
  { $set: { status: "active" } }
)
```
