## 🎯 Overview

**Purpose**: Multi-channel notification system for task updates, family activities, reminders, and system events with BullMQ queue processing and Redis caching.

**Business Value**: Keeps users informed about task assignments, deadlines, family activities, and system updates. Supports in-app, email, push, and SMS delivery channels. Essential for user engagement and task completion tracking.

**Users**:
- **All Users** (Business, Child, Common) - Receive and manage notifications
- **Business Users** (Parents/Teachers) - Receive task and family notifications
- **Child Users** - Receive task assignments and reminders

## ✨ Key Features

- ✅ **Multi-Channel Delivery** - In-app, email, push notifications, SMS
- ✅ **BullMQ Queue Processing** - Async notification delivery with retry logic
- ✅ **Redis Caching** - Unread counts and recent notifications cached
- ✅ **Bulk Notifications** - Send to multiple users or roles at once
- ✅ **Scheduled Notifications** - Deliver at specific times (reminders)
- ✅ **Task Reminders** - Deadline-based reminders with customizable triggers
- ✅ **Notification Types** - Task, family, system, reminder, mention, assignment, deadline
- ✅ **Priority Levels** - Low, normal, high, urgent
- ✅ **Read/Unread Tracking** - Mark as read, unread count badges
- ✅ **Soft Delete** - Notifications preserved (not permanently deleted)

## 🏗️ Architecture Design

### Design Principles

1. **Async-First**: All notifications queued via BullMQ for async processing
2. **Multi-Channel**: Support in-app, email, push, SMS delivery
3. **Cache-Optimized**: Unread counts and recent notifications cached in Redis
4. **Scheduled Delivery**: Support future-dated notifications (reminders)
5. **Bulk Operations**: Efficient bulk notification sending (up to 1000 users)

### Tech Stack
- **Database**: MongoDB (Mongoose)
- **Cache**: Redis (unread counts, recent notifications)
- **Queue**: BullMQ (async notification processing)
- **Validation**: Zod
- **Auth**: JWT (from auth module)

## 🔒 Security Notes

> **⚠️ Compliance & Security Considerations**

### Data Protection
- ✅ **Soft Delete**: Notifications are never hard deleted, only marked as `isDeleted`
- ✅ **Access Control**: Users can only access their own notifications
- ✅ **Audit Trail**: `readAt`, `deliveredAt` track notification lifecycle

### Access Control
- ✅ **Role-Based Access**: All authenticated users can manage their notifications
- ✅ **Rate Limiting**: Strict limits prevent spam (10/min for send, 100/min for read)
- ✅ **JWT Authentication**: All endpoints require valid tokens
- ✅ **User Validation**: Verify notification ownership before operations

### Notification Security
- ✅ **Bulk Limit**: Maximum 1000 notifications per bulk request (prevents abuse)
- ✅ **Channel Validation**: Only valid channels accepted (in_app, email, push, sms)
- ✅ **Type Validation**: Only valid notification types accepted
- ✅ **Priority Validation**: Only valid priority levels accepted

### Queue Security
- ✅ **Retry Logic**: 3 attempts with exponential backoff
- ✅ **Error Handling**: Failed jobs logged but don't block processing
- ✅ **Job Removal**: Automatic cleanup on complete/fail

## 🗄️ Redis Caching

<details>
<summary>📦 <b>Click to view Redis Caching Details</b></summary>

| Key Pattern | TTL | Use Case | Where to Create/Invalidate |
|-------------|-----|----------|---------------------|
| `notification:user:{userId}:unread-count` | 10 min | Cache unread notification count | Created in `getUnreadCount()`; Invalidated in `markAsRead()`, `markAllAsRead()`, `deleteNotification()`, `createNotification()` |
| `notification:user:{userId}:notifications` | 5 min | Cache recent notifications (first page) | Created in `getUserNotifications()` (first page only); Invalidated in `markAsRead()`, `markAllAsRead()`, `deleteNotification()`, `createNotification()` |

### Cache Strategy

**Pattern**: Cache-aside (lazy loading)

**On Read**:
1. Check Redis cache
2. If hit → return cached data
3. If miss → read from database
4. Cache result (first page only for notifications)
5. Return data

**On Write**:
1. Perform database operation
2. Invalidate related cache keys
3. Return response

</details>

## ⚡ Events & Queues

<details>
<summary>⚡ <b>Click to view Events & Queues Details</b></summary>

### BullMQ Queues
- **Queue Name**: `notificationQueue`
  - **Job**: `sendNotification`
  - **Attempts**: 3
  - **Backoff**: Exponential (2s delay)
  - **RemoveOnComplete**: 100 jobs
  - **RemoveOnFail**: 500 jobs

### Socket Events
- **emits**: `notification:created` - When a new notification is created (to recipients)

### Email Events (Async)
- **Sends**: Email notifications (if channel includes 'email')
- **Pattern**: Queued via BullMQ (doesn't block response)

### Push Notifications (Async)
- **Sends**: Push notifications via Firebase FCM (if channel includes 'push')
- **Pattern**: Queued via BullMQ (doesn't block response)

## 🐛 Troubleshooting

<details>
<summary>🐛 <b>Click to view Troubleshooting Details</b></summary>

### Issue: Notifications not appearing

**Symptom**: User doesn't see new notifications

**Solution**: Check cache invalidation and query filters

```bash
# Check Redis cache
redis-cli
GET notification:user:userId:notifications

# Clear cache if needed
redis-cli
DEL notification:user:userId:notifications

# Check database
db.notifications.find({ receiverId: ObjectId("userId"), isDeleted: false })
```

---

### Issue: Unread count not updating

**Symptom**: Badge count shows old number

**Solution**: Check cache invalidation on mark as read

```bash
# Check unread count cache
redis-cli
GET notification:user:userId:unread-count

# Clear cache
redis-cli
DEL notification:user:userId:unread-count

# Verify database count
db.notifications.countDocuments({
  receiverId: ObjectId("userId"),
  status: { $ne: "read" },
  isDeleted: false
})
```

---

### Issue: Notifications not being delivered

**Symptom**: Notifications created but not delivered

**Solution**: Check BullMQ queue status

```bash
# Check Redis queue
redis-cli
LLEN bull:notificationQueue

# Check failed jobs
redis-cli
LLEN bull:notificationQueue:failed

# Check queue logs
tail -f logs/error.log | grep "notification"
```

---

### Issue: Rate limit exceeded

**Symptom**: Error message "Too many notification attempts"

**Solution**: Wait for rate limit to reset (1 minute)

```bash
# Check rate limit keys
redis-cli
KEYS rate_limit:notification-*

# Manually clear if needed
redis-cli
DEL rate_limit:notification-send:USER_IP
```

---

### Issue: Reminder not triggering

**Symptom**: Task reminder not sent at scheduled time

**Solution**: Check scheduled notification and queue

```bash
# Check reminder exists
db.taskreminders.findOne({ _id: ObjectId("reminder123") })

# Check scheduled notifications in queue
redis-cli
ZRANGE bull:notificationQueue:delayed 0 -1

# Check worker is running
ps aux | grep bull
```
