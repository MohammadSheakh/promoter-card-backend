## 🎯 Overview

**Purpose**: Complete authentication and authorization system with user registration, login, OAuth integration, password management, and session handling.

**Business Value**: Provides secure access control for the entire platform, enabling user onboarding, secure authentication, and account management. Essential for all user interactions.

**Users**:
- **All Users** (Business, Child, Common) - Register, login, manage passwords
- **Business Users** (Parents/Teachers) - Register with business role
- **Child Users** - Created by parents or via invitation

## ✨ Key Features

- ✅ **User Registration** - Email/password registration with OTP verification
- ✅ **User Login** - JWT-based authentication with access + refresh tokens
- ✅ **OAuth Integration** - Google and Apple social login
- ✅ **Password Management** - Forgot password, reset password, change password
- ✅ **Email Verification** - OTP-based email verification using Redis
- ✅ **Session Management** - Redis-backed session caching (7 day TTL)
- ✅ **Token Rotation** - Refresh token rotation on every use
- ✅ **Rate Limiting** - Brute force protection (5 login attempts / 15 min)
- ✅ **Device Management** - FCM token registration for push notifications
- ✅ **Account Security** - Failed login attempts, account locking

## 🏗️ Architecture Design

### Design Principles

1. **Security-First**: Rate limiting, password hashing, JWT with short expiry
2. **Redis Caching**: Session caching, OTP storage, token blacklisting
3. **Token Rotation**: Refresh token rotation on every use for security
4. **Email Verification**: OTP-based verification before login
5. **OAuth Support**: Google and Apple login with automatic account linking

### Tech Stack
- **Database**: MongoDB (Mongoose)
- **Cache**: Redis (session storage, OTP, token blacklisting)
- **Email**: Nodemailer (SendGrid/AWS SES)
- **Validation**: Zod
- **Auth**: JWT (access token: 15 min, refresh token: 7 days)
- **Password**: bcryptjs (12 rounds)
- **OAuth**: google-auth-library, apple-signin-auth

## 🔒 Security Notes

> **⚠️ Compliance & Security Considerations**

### Data Protection
- ✅ **Password Hashing**: bcrypt with 12 rounds (industry standard)
- ✅ **JWT Tokens**: Short-lived access tokens (15 min) + refresh tokens (7 days)
- ✅ **Token Blacklisting**: Revoked tokens stored in Redis until expiry
- ✅ **Email Verification**: Required before login (prevents fake accounts)

### Access Control
- ✅ **Rate Limiting**: Strict limits prevent brute force (5 login attempts / 15 min)
- ✅ **Account Locking**: Lock after max failed attempts (configurable)
- ✅ **Role-Based Access**: All endpoints require valid JWT tokens
- ✅ **Session Management**: Redis-backed sessions with 7 day TTL

### OAuth Security
- ✅ **Token Verification**: Google/Apple tokens verified before login
- ✅ **Email Linking**: Auto-link OAuth to existing local accounts (if email verified)
- ✅ **Provider Isolation**: Separate OAuth accounts per provider

### OTP Security
- ✅ **Redis Storage**: OTPs stored in Redis with 10 min TTL
- ✅ **Rate Limiting**: Max 3 OTPs per hour per email
- ✅ **Attempt Limiting**: Max 5 verification attempts per OTP
- ✅ **Cooldown Period**: 60 second cooldown between OTP requests

### Service: Register User

**Figma-Flow-Screenshot-File Name**: (Integrated in all onboarding flows)

**Endpoint**: `POST /auth/register`  
**Role**: Public (New User)  
**Auth**: None  
**Rate Limit**: 10 requests/hour (strict)  
**Middleware**: `registerLimiter`, `AuthController.register`

<details>
<summary>🔄 <b>Click to view Flow Diagram</b></summary>

```
New User
    ↓
POST /auth/register
    ↓
Controller: register()
    ↓
Service: createUser()
    ├─→ Check if email already exists
    ├─→ If exists and verified → reject
    ├─→ If exists and not verified → update and resend OTP
    ├─→ Hash password (bcrypt 12 rounds)
    ├─→ Create UserProfile (acceptTOC: true)
    ├─→ Create User account
    ├─→ Create verification token
    ├─→ Send OTP via email (Redis-based)
    └─→ Return { user, verificationToken }
    ↓
Response: 201 Created
```

</details>

**Redis Cache Keys Created**:
- `otp:verify:{email}` (TTL: 10 min) → Created in `sendVerificationOtp()` for email verification
- `otp:send_count:{email}` (TTL: 1 hour) → Created in `sendVerificationOtp()` for rate limiting

## 🗄️ Redis Caching

<details>
<summary>📦 <b>Click to view Redis Caching Details</b></summary>

| Key Pattern | TTL | Use Case | Where to Create/Invalidate |
|-------------|-----|----------|---------------------|
| `session:{userId}:{fcmToken}` | 7 days | Cache user sessions | Created in `login()`, `loginV2()`; Invalidated in `logout()`, `forgotPassword()`, `resetPassword()`, `changePassword()` |
| `otp:verify:{email}` | 10 min | Store verification OTPs | Created in `sendVerificationOtp()`; Invalidated in `verifyOtp()` |
| `otp:reset_password:{email}` | 10 min | Store reset password OTPs | Created in `sendResetPasswordOtp()`; Invalidated in `verifyResetPasswordOtp()` |
| `otp:cooldown:{email}` | 60 sec | Cooldown between OTP requests | Created in `sendVerificationOtp()`; Invalidated in `verifyOtp()` |
| `otp:send_count:{email}` | 1 hour | Track OTP send count | Created in `sendVerificationOtp()`; Auto-expires |
| `blacklist:{refreshToken}` | Matches token expiry | Blacklist revoked tokens | Created in `logout()`, `refreshAuth()`; Auto-expires |

### Cache Strategy

**Pattern**: Cache-aside (lazy loading)

**On Authentication**:
1. Check Redis session cache
2. If hit → return cached session data
3. If miss → authenticate from database
4. Cache session (7 days)
5. Return data

**On Logout/Password Change**:
1. Perform database operation
2. Invalidate all related cache keys
3. Blacklist tokens
4. Return response

</details>

## 🐛 Troubleshooting

<details>
<summary>🐛 <b>Click to view Troubleshooting Details</b></summary>

### Issue: Can't login - email not verified

**Symptom**: Error message "Please verify your email before logging in"

**Solution**: Check email for verification OTP or request resend

```bash
# Check user's email verification status
db.users.findOne({ email: "test@example.com" }, { isEmailVerified: 1 })

# Resend verification OTP
curl -X POST http://localhost:5000/auth/resend-otp \
  -H "Content-Type: application/json" \
  -d '{ "email": "test@example.com" }'
```

---

### Issue: Too many login attempts

**Symptom**: Error message "Too many login attempts, please try again later"

**Solution**: Wait 15 minutes for rate limit to reset

```bash
# Check rate limit keys in Redis
redis-cli
KEYS rate_limit:login:*

# Manually clear rate limit (if needed)
redis-cli
DEL rate_limit:login:USER_IP
```

---

### Issue: Session not cached

**Symptom**: Login successful but session not found in Redis

**Solution**: Check Redis connection and session creation

```bash
# Check Redis connection
redis-cli ping  # Should return: PONG

# Check session keys
redis-cli
KEYS session:*

# Check session TTL
redis-cli
TTL session:userId:fcmToken
```

---

### Issue: Token refresh fails

**Symptom**: Error message "Refresh token has been revoked"

**Solution**: Token was blacklisted (security feature) - user must login again

```bash
# Check if token is blacklisted
redis-cli
GET blacklist:refreshToken

# Token blacklisting is intentional for security
# User must login again with credentials
```

---

### Issue: OTP not received

**Symptom**: User doesn't receive verification/reset OTP email

**Solution**: Check SMTP configuration and OTP send count

```bash
# Check SMTP configuration
cat .env | grep SMTP

# Check OTP send count
redis-cli
GET otp:send_count:test@example.com

# Check error logs
tail -f logs/error.log | grep "OTP"
```

</details>
