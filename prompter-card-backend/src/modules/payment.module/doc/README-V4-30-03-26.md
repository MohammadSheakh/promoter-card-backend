## 🎯 Overview

**Purpose**: Provides comprehensive payment processing infrastructure with multi-gateway support (Stripe, RevenueCat, SSLCommerz), transaction tracking, webhook handling, and admin earnings dashboard.

**Business Value**: Enables monetization through secure payment processing, complete transaction audit trails, automatic subscription activation via webhooks, refund processing, and comprehensive earnings analytics for business intelligence.

**Users**:
- **Individual Users** - Make payments for subscriptions, purchases, and services via web/mobile
- **Business Users** - Connect Stripe accounts for receiving payments (Express accounts)
- **Admins** - View earnings dashboard, manage transactions, process refunds, monitor payment health

## 🏗️ Architecture Design

### Design Principles

1. **Strategy Pattern** - Pluggable payment gateways and purchase strategies
2. **Webhook-Driven** - Real-time payment status updates via webhooks
3. **Idempotency** - Prevent duplicate transactions with payment intent/order ID checks
4. **Reference Tracking** - Generic reference system links payments to any entity
5. **Multi-Gateway Abstraction** - Unified interface for different payment providers

### Tech Stack
- **Database**: MongoDB (Mongoose) with pagination support
- **Cache**: Redis (cache-aside pattern for earnings overview)
- **Payment Gateways**: 
  - Stripe (web subscriptions, one-time payments, Connect)
  - RevenueCat (mobile in-app purchases iOS/Android)
  - SSLCommerz (regional payment gateway)
- **Validation**: Zod schemas
- **Auth**: JWT with role-based access control (TRole)
- **Notifications**: Web notifications via `enqueueWebNotification`

## 🔒 Security Notes

> **⚠️ Compliance & Security Considerations**

### Payment Card Industry (PCI) Compliance
- ✅ **No Card Data Storage**: Card details never stored in database (handled by Stripe/RevenueCat)
- ✅ **Secure Webhooks**: HMAC-SHA256 signature verification for all webhook events
- ✅ **Tokenization**: Payment intents and transaction IDs used for references
- ✅ **Stripe Express**: PCI-compliant connected account onboarding

### Data Protection
- ✅ **Soft Delete**: Transactions preserved with `isDeleted` flag (audit compliance)
- ✅ **Complete Audit Trail**: PaymentTransaction records maintain full payment history
- ✅ **Idempotency**: Webhook handlers prevent duplicate transactions via payment intent checks
- ✅ **Gateway Response Storage**: Full gateway responses stored for dispute resolution

### Access Control
- ✅ **Role-Based Access**: 
  - `admin` - Full transaction viewing, earnings dashboard, refund processing
  - `common` - Basic payment operations
  - `business` - Stripe Connect account creation
- ✅ **User Isolation**: Users can only view their own transactions (filtered by `userId`)
- ✅ **Webhook Security**: Signature verification prevents fake webhook attacks

### Webhook Security
- ✅ **Signature Verification**: Both Stripe and RevenueCat webhooks verify HMAC signatures
- ✅ **Timing-Safe Comparison**: Prevents timing attacks on signature validation
- ✅ **Raw Body Middleware**: `express.raw()` positioned before `express.json()` for signature integrity
- ✅ **Secret Validation**: Missing webhook secrets return 500 errors (fail closed)

### Fraud Prevention
- ✅ **Transaction Validation**: SSLCommerz validation API integration
- ✅ **Payment Intent Verification**: Stripe payment intents verified before activation
- ✅ **Reference Integrity**: Database references validated before transaction creation
- ✅ **Failed Payment Logging**: Failed webhooks logged for investigation

## 🗄️ Redis Caching

<details>
<summary>📦 <b>Click to view caching details</b></summary>

| Key Pattern | TTL | Use Case | Where to Invalidate |
|-------------|-----|----------|---------------------|
| `payment:transactions:admin` | 5 min | Cache admin transaction list | Webhook handlers, refund operations |
| `payment:earnings:overview` | 5 min | Cache earnings dashboard | Transaction status changes, refunds |
| `payment:transaction:{id}` | 10 min | Cache individual transaction | Transaction updates |
| `payment:user:{userId}:transactions` | 10 min | Cache user transaction history | New transaction creation |

### Cache Strategy

**Pattern**: Cache-aside (lazy loading)

**On Read**:
1. Check Redis cache for key
2. If hit → return cached data
3. If miss → read from database
4. Cache result with TTL
5. Return data

**On Write**:
1. Perform database operation (create/update transaction)
2. Invalidate related cache keys
3. Return response

</details>

### Socket Events

### Webhook Events (Async)

#### Stripe Webhooks (`/api/v1/stripe-webhook`)
- `checkout.session.completed` - One-time payment completed → Create PaymentTransaction
- `invoice.payment_succeeded` - Recurring payment/trial conversion → Update UserSubscription
- `payment_intent.payment_failed` - Payment failed → Log failure, notify user
- `checkout.session.expired` - Session expired → Log and notify
- `customer.subscription.created` - Subscription created → Capture dates
- `customer.subscription.deleted` - Subscription cancelled → Update status
- `customer.subscription.trial_will_end` - Trial ending soon → Notify user (3 days before)
- `transfer.created` - Stripe Connect transfer → Update booking status

#### RevenueCat Webhooks (`/api/v1/revenuecat-webhook`)
- `INITIAL_PURCHASE` - First purchase → Create UserSubscription + PaymentTransaction
- `RENEWAL` - Subscription renewed → Update dates, create PaymentTransaction
- `CANCELLATION` - User cancelled → Set cancelledAtPeriodEnd
- `EXPIRATION` - Subscription expired → Set status to expired
- `REFUND` - Refund processed → Set payment status, cancel subscription
- `BILLING_ISSUE` - Payment failed → Set status to past_due
- `SUBSCRIPTION` - General update → Sync subscription state

### Notification Events
- **Payment Success** - Web notification sent after successful payment
- **Payment Failed** - Web notification sent after payment failure
- **Subscription Activated** - Web notification sent after subscription activation
- **Trial Ending** - Web notification sent 3 days before trial ends

## 🐛 Troubleshooting

<details>
<summary>🔧 <b>Click to view troubleshooting guides</b></summary>

### Issue: Webhook Signature Verification Failed

**Symptom**: Webhook returns 400/401, logs show "Webhook signature verification failed"

**Solution**: Verify webhook secret and middleware order

```bash
# Check webhook secret in .env
cat .env | grep STRIPE_WEBHOOK_SECRET
cat .env | grep REVENUECAT_WEBHOOK_SECRET

# Verify express.raw() is before express.json() in app.ts
grep -A 5 "stripe-webhook" src/app.ts
grep -A 5 "revenuecat-webhook" src/app.ts
```

**Fix**: Ensure middleware order:
```typescript
// ✅ CORRECT
app.post('/api/v1/stripe-webhook', express.raw({ type: 'application/json' }), handler);
app.use(express.json());

// ❌ WRONG
app.use(express.json());
app.post('/api/v1/stripe-webhook', handler);  // Will fail!
```
