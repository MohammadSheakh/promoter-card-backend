## 1. Executive Summary

### 1.1 Product Overview

**PromoterCard** is a comprehensive web platform for businesses that rely on promoters to engage potential customers without using traditional marketing methods like flyers. The system provides promoters with custom NFC cards that link to personalized landing pages for each event. On these landing pages, users can interact with simple games (dice roll, spin wheel) to win prizes. The platform tracks key metrics including page visits, sign-ups, and Google reviews generated by promoters.

### 1.2 Deployment Model

**Single-Business Deployment:**
- One PromoterCard instance = One business
- No multi-tenancy
- No business isolation required
- Simplified authentication and authorization
- No subscription/billing management
- No super admin role

**Admin manages:**
- Their own promoters
- Their own campaigns and events
- Their own landing pages
- Their own NFC cards
- All analytics and reports

## 2. Technology Stack

### 2.1 Core Stack

```yaml
Runtime: Node.js v20.x (LTS)
Language: TypeScript v5.x (strict mode)
Framework: Express.js v4.x
Database: MongoDB v7.x
ODM: Mongoose v8.x
Cache: Redis v7.x (cache-aside pattern)
Queue: BullMQ v5.x (async operations)
Validation: Zod v3.x (100% coverage)
Auth: JWT (short expiry) + Refresh Token rotation
Security: Helmet.js, CORS whitelist, NoSQL injection sanitization
Logging: Winston (structured JSON)
Testing: Jest + Supertest
Pagination: Custom PaginationService
Architecture: Modular, SOLID, stateless, horizontally scalable
```

### 2.2 Generic Patterns

```yaml
Generic Controller:
  - getAllWithPaginationV2
  - getAllWithAggregation
  - getById
  - create
  - update
  - delete

Generic Service:
  - CRUD operations
  - Pagination handling
  - Aggregation pagination
  - Error handling

Middleware (from middleware/ folder):
  - authenticate
  - authorize
  - validateRequest
  - sanitizeInput
  - rateLimiter (Redis-backed, sliding window)
  - requestLogger
  - errorHandler
```

### 3.2 Key Design Decisions

**Single-Tenant (No Multi-Tenancy):**
- No `businessId` on collections
- No business scoping middleware
- Single admin user (or multiple admin users for the same business)
- Simplified authorization

**Generic Controllers/Services:**
- DRY principle applied
- Consistent CRUD patterns
- Pagination handled generically
- Aggregation pagination via PaginationService

**Cache-Aside Pattern:**
- Read from cache first
- On miss вҶ’ read DB вҶ’ cache вҶ’ return
- On write вҶ’ invalidate cache

**BullMQ for Heavy Operations:**
- All operations > 500ms вҶ’ async
- Report generation вҶ’ BullMQ
- Analytics aggregation вҶ’ BullMQ
- Email sending вҶ’ BullMQ

### 6.1 User Schema

```typescript
import { Schema, model, Types, Document } from 'mongoose';

export enum UserRole {
  ADMIN = 'admin',
  CAMPAIGN_MANAGER = 'campaign_manager',
  PROMOTER = 'promoter',
}

export type TUserRole = `${UserRole}`;

export interface IUser extends Document {
  email: string;
  passwordHash: string;
  name: string;
  role: TUserRole;
  status: 'active' | 'inactive' | 'suspended';
  emailVerified: boolean;
  lastLoginAt?: Date;
  createdAt: Date;
  updatedAt: Date;
}

const userSchema = new Schema<IUser>(
  {
    email: {
      type: String,
      required: true,
      unique: true,
      lowercase: true,
      trim: true,
    },
    passwordHash: {
      type: String,
      required: true,
    },
    name: {
      type: String,
      required: true,
      trim: true,
    },
    role: {
      type: String,
      enum: Object.values(UserRole),
      default: UserRole.ADMIN,
    },
    status: {
      type: String,
      enum: ['active', 'inactive', 'suspended'],
      default: 'active',
    },
    emailVerified: {
      type: Boolean,
      default: false,
    },
    lastLoginAt: {
      type: Date,
    },
  },
  {
    timestamps: true,
  }
);

// Indexes
userSchema.index({ email: 1 }, { unique: true });
userSchema.index({ role: 1, status: 1 });

export const User = model<IUser>('User', userSchema);
```

### 8.1 Critical Indexes

| Collection | Index | Type | Purpose |
|------------|-------|------|---------|
| User | `{ email: 1 }` | Unique | Login queries |

// Cache key format: <module>:<identifier>:<datatype>

### 9.2 Cache Keys & TTLs

| Data Type | Cache Key Pattern | TTL | Rationale |
|-----------|-------------------|-----|-----------|
| User profile | `user:profile:{userId}` | 15 min | Moderate freshness |
| Landing page (public) | `landingpage:public:{slug}` | 5 min | Published, stable |
| Landing page list | `landingpage:list:{page}` | 2 min | Frequent edits |
| Promoter profile | `promoter:profile:{promoterId}` | 15 min | Moderate freshness |
| Promoter leaderboard | `promoter:leaderboard` | 5 min | Near real-time |
| Card details | `card:detail:{cardId}` | 15 min | Moderate freshness |
| NFC UUID resolution | `card:nfc:{nfcUuid}` | 1 hour | Static mapping |
| Campaign list | `campaign:list:{status}` | 2 min | Moderate updates |
| Upcoming events | `event:upcoming` | 5 min | Near real-time |
| Participant count | `participant:count` | 2 min | Frequent updates |
| Game statistics | `game:stats:{period}` | 2 min | Near real-time |
| Prize config | `prize:config:{landingPageId}` | 15 min | Rarely changes |
| Review statistics | `review:stats:{period}` | 2 min | Frequent updates |
| Social link list | `sociallink:list:{landingPageId}` | 15 min | Rarely changes |
| Social analytics | `sociallink:analytics:{period}` | 5 min | Near real-time |
| Dashboard metrics | `analytics:overview:{period}` | 1 min | High-frequency |
| Report list | `report:list:{status}` | 1 min | Frequent updates |
| Session | `session:{userId}:{deviceId}` | 7 days | Match refresh token |

### 9.3 Cache Invalidation

**On write operations:**
- Create: Invalidate related list caches
- Update: Invalidate detail cache + related list caches
- Delete: Invalidate detail cache + related list caches

**Example:**
```typescript
// On landing page update
await invalidateCache([
  `landingpage:detail:${id}`,
  `landingpage:public:${slug}`,
  `landingpage:list:*`,
]);
```
